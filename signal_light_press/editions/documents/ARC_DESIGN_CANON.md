Scope: Physical Architecture — Arc Design
I. GOVERNING PRINCIPLE



Routes are lines.

Arcs are fields.



A route assumes continuity.

An Arc assumes disruption and survives anyway.



II. DEFINITION — SIGNAL ARC



A Signal Arc is a bounded system of movement and storage that:



Links multiple nodes (ports, depots, communities)



Operates on seasonal rhythm



Maintains redundant alternatives



Has graceful degradation as default behavior



Treats storage as primary, transport as secondary



Keeps human sovereignty at the decision edge



III. ARC REQUIRED ELEMENTS (NON-NEGOTIABLE)



Every canonical Arc must have all of the following:



1\. Nodes (Minimum: 2)



At least two distinct nodes capable of independent operation.



A node is not “a stop.”

A node is a place where the Arc can pause without failing.



2\. Cold / Holding Capability (Minimum: 1 per Arc)



At least one node must provide:



cold storage, or



durable holding, or



provisioning reserves



Without holding capability, you have a delivery route — not an Arc.



3\. Seasonal Rhythm Declaration



Each Arc must declare its seasonal mode:



Active



Reduced



Dormant



Emergency-only



Arcs may sleep.

Sleeping is not failure.



4\. Redundancy Path



Each Arc must define at least one of:



Alternate port/node



Alternate land shuttle corridor



Alternate holding strategy



Redundancy is required, not aspirational.



5\. Failure Posture



Each Arc must include a defined failure response hierarchy:



Hold (storage-first)



Reroute (redundant node)



Downshift (reduced service)



Sleep (dormant posture)



Deploy (emergency-only mode)



IV. ARC STATES (CANONICAL)



An Arc may exist in one of the following states:



Resonant / Open — movement windows are favorable



Stable / Consolidating — movement allowed, no expansion



Restricted / Narrow Window — only essential movements



Dormant / Holding — storage posture dominates



Emergency / Surge — governed deployment mode



State transitions must be consistent with:



HAL (alignment)



G(t) (trust gate)



Memory stack (Ghost/Afterglow drift signals)



Human constraints



V. ARC CLASSIFICATIONS (TYPES)

Type A — Coastal Arc



Ports as primary nodes; vessel as mobile node.



Type B — River Arc



River nodes + shallow draft rhythm + seasonal constraints.



Type C — Hybrid Arc



Sea + inland depot(s) + shuttle corridors.



Type D — Emergency Arc



Pre-staged provisions + deployment-first operating logic.



Each Arc must declare a type.



VI. CANONICAL DESIGN RULES



No single point of failure



Storage precedes transport



Nodes are cultural as well as functional



Arcs must be able to sleep



Human arbitration is always final



Expansion is a separate ritual (requires doctrine approval)



VII. ACCEPTANCE TEST (IS IT REALLY AN ARC?)



An Arc is canonical if it can answer “yes” to all:



Can it continue functioning if one node fails?



Can it hold goods without immediate delivery?



Does it have declared seasonal posture?



Does it have at least one redundancy strategy?



Can it downshift gracefully without breaking promises?



If no, it is not an Arc.



VIII. CANONICAL RULE



An Arc is a promise to endure disruption without panic.
